Special Token Injection (STI) Completed 2026

A new vulnerability class against LLM inference providers and agentic systems

Premise

Modern LLMs use a small set of special tokens (e.g. role markers, beginning/end-of-turn, system delimiters) to structure conversations and tool calls. When a provider, agent framework, or downstream application accepts user-controlled input that is later tokenized without sanitization, an attacker can inject those special tokens directly. The model then interprets the injected tokens as authoritative structural signals, breaking trust boundaries that were assumed to be enforced at the prompt level.

Special Token Injection (STI) generalizes this class of attack across providers, frameworks, and applications. It is distinct from prompt injection: prompt injection persuades a model with text, while STI rewrites the conversation’s structural grammar.

Why a new vulnerability category

  • Existing categories (Prompt Injection, Insecure Output Handling) describe the symptom space, not the structural root cause.
  • STI is detectable, fixable, and testable with concrete primitives at the tokenization layer, which makes it amenable to standardization.
  • Impact spans confidentiality, integrity, and availability: role hijacking, system-prompt overwrite, tool-call forgery, and data exfiltration through agent loops.
Standardization status: A formal CWE entry for this vulnerability class is currently under review: CWE-CAPEC submission ES2512.

Key findings

  • Validated STI on major inference providers evaluated, with provider-specific tokenizer quirks producing distinct exploit primitives.
  • Built a working PoC chaining STI in Grok to trigger Server-Side Request Forgery (SSRF) and exfiltrate an X API token from the inference environment.
  • Identified consistent failure modes: input-side sanitization missing entirely, partial denylists, inconsistent treatment between text and tool-call payloads.
