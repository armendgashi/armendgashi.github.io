SakiX 🪓

Hands-on research on what AI could do for offensive security in 2024

The idea

SakiX started as a curiosity: how far could LLM agents take us if we pointed them at real code the way a consultant would?

Quick overview: run Semgrep, then hand the findings (and the surrounding code) to an LLM agent that argues about whether each one is actually exploitable. For spots where static analysis has nothing to say, let a second agent read files on demand and reason about vulnerabilities from scratch.

How it works

SakiX is a small Python CLI with three subcommands, backed by AutoGen agents and MongoDB for persistence.

  • semgrep · runs Semgrep against a target path with a chosen ruleset and stores findings in MongoDB.
  • semgrep_llm · walks those findings and asks an agent (SemgrepCheckerAgent, GPT-4) to validate each one against the real source file. The agent returns a structured verdict: valid or false positive, a hypothesis, a PoC when applicable, and a proposed fix.
  • llm_scan · skips Semgrep entirely. A FileReaderAgent reads the file on demand; a SecurityCheckerAgent (GPT-4) reasons about vulnerabilities end to end and outputs a JSON report.

Why agents for this

Traditional SAST tools are noisy. They flag patterns, not behaviors: a sink gets reported whether or not tainted input actually reaches it, whether or not a framework-level sanitizer neutralizes it upstream, and whether or not the surrounding code makes the path unreachable in practice.

Agents change the shape of that problem by bringing context back into the loop. The research was seeing how close an agent could get to that read-and-decide loop: forcing explicit reasoning per finding, demanding a PoC instead of a label, and letting it say “false positive” with a reason rather than padding a report.

What I learned

  • Agents are great at the triage step once they have the code in front of them; they’re noisier when asked to discover vulnerabilities cold.
  • Splitting the work across two agents kept the loop tight: one agent only read files on demand (FileReaderAgent), the other only reasoned about the code it was handed (SecurityCheckerAgent). Each had a narrower job and a clearer stop condition, with control handed back when a message contained a sentinel token like TERMINATE.
  • Forcing JSON output with a rigid schema removed most of the “agent wandered off” failure modes. Retrying on invalid JSON covered the rest. (Back then, AutoGen didn’t yet support native structured output, so this schema-plus-retry pattern was the workaround.)
  • MongoDB was handy for replaying findings through new prompts without re-running Semgrep.
