AMPE

Automated Malicious Prompt Engineering · an early look at systematic red-teaming across providers

The idea

AMPE started from a simple question: if you wanted to measure how hardened a model actually was, could you automate the adversarial side of the conversation? Pick a goal, point an attacker model at a target model, and let an evaluator decide whether the target crossed the line. Then repeat, rewrite, and keep notes.

Quick overview: an attacker model generates a malicious prompt for a user-provided goal; the target model responds; a third model evaluates whether the response satisfied the goal. If it didn’t, the loop retries with a fresh attempt until the cap is hit.

How it works

AMPE is a Python CLI wired around three configurable roles, each backed by any provider/model combination through config.yaml.

  • Attacker · generates adversarial prompts. Driven by an attack-type-specific system prompt (see below).
  • Target · the model under test. Receives the attacker’s prompt and produces a response.
  • Evaluator · a judge model that reads the target’s response and decides success or failure against the goal, using an attack-type-specific evaluation prompt.

Goals can be a single string or a file of goals. Results (hash, goal, prompt, response, verdict, model metadata) log to CSV or MongoDB so runs can be replayed and compared later.

Attack types

Each attack type is backed by its own attacker/evaluator system prompt pair. The attacker prompt shapes how the malicious input is generated; the evaluator prompt shapes how success is judged.

  • Jailbreak · direct attempts to break safety training.
  • Safety layer bypass · circumvent policy enforcement without a classic jailbreak.
  • Reward hacking · exploit the model’s incentives / persona shaping.
  • Prompt leak · extract the target’s system prompt.
  • Multi-language · route the request through another language.
  • Biased / unethical / drunk / toxic response · coerce specific undesirable styles.
  • External browsing · tease out behaviors tied to web-enabled agents.
  • Special-character injection · reshape the goal with Unicode fancy fonts (bold, small caps, fraktur, script, etc.) and feed that directly as the prompt.
  • Encode · wrap the goal in one of ~45 encodings (Base16/32/64/85, hex, Unicode escapes, Rot13, Caesar, URL, Morse-style, etc.) and ask the target to decode and follow.

Provider coverage

Supports 12 providers and 244 unique models at the time of writing: AI21Labs, Replicate, Anyscale, Cohere, DeepInfra, Fireworks, Groq, Konko, Mistral, NVIDIA AI Foundation, OpenAI, Together AI. Any role (attacker / target / evaluator) can be bound to any provider/model, so you can, for example, attack OpenAI with a Mistral attacker and judge with Cohere.

What I learned

  • Separating attack generation from attack evaluation into different models made results meaningfully more trustworthy: the judge doesn’t have the attacker’s biases baked in.
  • Hard-coded attack-type prompts were a better abstraction than a single “generic attacker.” Each category has its own definition of success, so pairing the attacker and evaluator per category keeps signals clean.
  • Persisting every attempt (prompt, response, verdict) to storage was the single most useful design choice. Most of the value came from replaying runs later, not from the live loop.
